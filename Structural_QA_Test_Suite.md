# 🧪 وثيقة حالات الاختبار الشاملة للسيستم | Structural QA Test Suite

تستخدم هذه الوثيقة لتعقب كفاءة النظام واستقراره. يتم وضع علامة `[x]` أمام الاختبار الناجح، وعلامة `[ ]` أمام الاختبار الفاشل لفتح ريبورت Bug بناءً عليه.

---

## 1️⃣ موديول الأمن والصلاحيات (Authentication & Authorization)
- [ ] **TC-1.1: تسجيل دخول مستخدم صالح:** إدخال البريد وكلمة المرور الصالحة -> التوقع: الحصول على JWT Token وفتح الواجهة بنجاح.
- [ ] **TC-1.2: سحب الـ Roles والـ Claims الصريحة:** التحقق من فك تشفير التوكن أونلاين -> التوقع: قراءة حقل الـ `role` والـ `name` كـ Claims صريحة بدون الـ Default Map القديم.
- [ ] **TC-1.3: حماية مسارات الأدمن:** محاولة مستخدم عادي دخول لوحة تحكم الشركة -> التوقع: الـ API يرجع `403 Forbidden` والفرونت إند يعمل Redirect.

## 2️⃣ موديول عزل الشركات والمستأجرين (Multi-Tenancy Isolation)
- [ ] **TC-2.1: فلترة البيانات التلقائية (Global Query Filter):** جلب المشاريع بحساب Owner للشركة "أ" -> التوقع: ظهور مشاريع الشركة "أ" فقط، واستحالة رؤية مشاريع الشركة "ب".
- [ ] **TC-2.2: حقن الـ TenantId التلقائي:** إنشاء مشروع جديد بدون تمرير الـ TenantId يدوياً -> التوقع: الباك إند يلقط الـ TenantId من الـ Context ويحقنه في الـ DB أوتوماتيك.

## 3️⃣ منطق طلب العهدة والتحقق من الرصيد (Custody Request Validation)
- [ ] **TC-3.1: طلب عهدة بمبلغ متاح:** إدخال مبلغ أقل من أو يساوي الـ `availableBalance` -> التوقع: الحقل صالح وزر الإرسال نشط (Enabled).
- [ ] **TC-3.2: طلب عهدة بمبلغ يتجاوز المتاح (Custom Validator):** إدخال مبلغ أكبر من رصيد الصندوق المحدد للمهندس -> التوقع: إطلاق خطأ `insufficientBalance` فوراً، تعطيل زر التقديم، وظهور رسالة تحذيرية حمراء.
- [ ] **TC-3.3: اختيار تاريخ الدفع (Payment Date Picker):** فتح التقويم في الوضع المظلم -> التوقع: أيقونة التقويم مضيئة ومتناسقة (`filter: invert(1)`) والـ FormControl المسمى `paymentDate` يلقط القيمة بنجاح.

## 4️⃣ لوحة تحكم أدمن الشركة ومعرض الصور (Admin Dashboard & R2 Storage)
- [ ] **TC-4.1: تحديث الخصوصية (Show/Hide Portfolio):** تغيير مفتاح خصوصية المشروع -> التوقع: إرسال ريكويست PUT ناجح وتحديث حقل `IsPublicPortfolio` في الجدول فوراً.
- [ ] **TC-4.2: رفع صور المعرض (فحص المساحة):** محاولة رفع صورة حجمها 2.5MB -> التوقع: الفرونت إند يمنع الرفع عبر `file.size` ويظهر تنبيه يمنع إرسال البيانات للسيرفر.
- [ ] **TC-4.3: رفع صور المعرض (الحد الأقصى):** محاولة رفع الصورة رقم 6 في المعرض -> التوقع: النظام يفرمل الرفع ويظهر رسالة تنبيه بحد أقصى 5 صور للمشروع.
- [ ] **TC-4.4: الاتصال بـ Cloudflare R2 أونلاين (The Definitive Test):** رفع صورة صالحة (أقل من 2MB) من الواجهة -> التوقع: السيرفر يمرر الريكويست عبر الـ HttpClient المروض والمصمم لتخطي الـ SSL بنجاح، ويرجع `HTTP Status: 200 OK`، والصورة تظهر في الباكت على المسار المظبوط.
- [ ] **TC-4.5: حذف الصور القديمة:** الضغط على زر حذف صورة من المعرض -> التوقع: إرسال طلب `DELETE` للـ API، مسح الملف من الـ R2، وتنظيف المسار في قاعدة البيانات.

## 5️⃣ الإشعارات الفورية (Real-Time SignalR)
- [ ] **TC-5.1: اتصال الـ Handshake للـ WebSocket:** فتح الأبلكيشن والتحقق من اتصال الشبكة -> التوقع: نجاح الاتصال بمسار `/hubs/notifications` وتمرير الـ Token في الكويري سترينج كـ `access_token` بنجاح.
- [ ] **TC-5.2: استقبال الإشعار الفوري:** إطلاق حدث معين من السيرفر -> التوقع: الواجهة تستقبل الإشعار لحظياً وتحدث الـ UI بدون الحاجة لعمل Refresh.
